# GatherUp v0.1 账号系统设计

## 1. 当前产品判断

GatherUp 不区分“参与者账号”和“组织者账号”。

一个用户只需要一个账号。登录后，这个账号可以：

- 浏览活动。
- 参与数调和地点投票。
- 报名活动。
- 查看历史订单。
- 创建活动。
- 管理自己创建的活动。

“参与者”和“组织者”不是账号类型，而是用户在某个活动里的关系。

## 2. 当前原型状态

当前 Web App 使用演示账号和浏览器本地账号模拟登录：

```text
邮箱：miki@gatherup.local
密码：gatherup123
```

已具备的原型能力：

- 邮箱密码登录。
- 本地注册账号。
- 原型验证码登录入口。
- 忘记密码入口。
- Supabase Auth 适配层。
- Supabase 用户资料同步准备。
- 首次资料补全页。
- 按账号邮箱记录资料补全状态。
- 登录状态暂时保存在浏览器 cookie 中。
- 本地注册账号暂时保存在浏览器 localStorage 中。

未配置 Supabase 环境变量时，这些能力仍是原型能力，用于验证页面和流程，不是正式账号系统。

配置 Supabase 后：

- 邮箱密码登录、验证码登录和找回密码会走 Supabase Auth。
- 登录成功后会创建或读取 `users` 表中的 GatherUp 业务用户。
- `users.auth_user_id` 会绑定 Supabase Auth 的 `auth.users.id`。
- `users.public_id` 会成为活动、订单、同行人填写和组织者权限识别的 GatherUp ID。

正式上线前必须替换为后端账号服务，不能依赖 localStorage 保存账号或密码。

## 3. 正式账号目标

正式账号系统需要满足：

- 用户长期有效。
- 换设备后可以找回账号。
- 可以绑定邮箱、Google、Apple、手机号、微信等多种登录方式。
- 可以使用验证码完成注册、登录、找回密码、换绑。
- 密码不能明文保存。
- 登录状态不能只依赖前端 cookie。
- 用户的活动、订单、付款记录、座位和通知都绑定到稳定的 `user_id`。
- 全球用户可以不依赖某个地区平台完成注册和登录。

## 4. 推荐登录方式

v0.1 建议按优先级推进：

1. 邮箱验证码登录。
2. 邮箱密码登录。
3. Google 登录。
4. Apple 登录。
5. 手机号验证码登录。
6. 微信登录。

原因：

- 邮箱验证码成本低，适合早期测试，也是全球最通用的账号底座。
- 邮箱密码方便固定测试账号和长期登录。
- Google 和 Apple 覆盖全球用户，适合作为快捷登录。
- 手机号验证码体验好，但有短信成本和合规成本。
- 微信登录适合中国用户和未来小程序场景，但不适合作为全球主入口。

## 5. 全球账号策略

GatherUp 的主身份应该是平台自己的 `user_id`，而不是某个第三方平台账号。

推荐模型：

- `user_id`：GatherUp 内部永久用户 ID。
- `email_identity`：邮箱验证码或邮箱密码登录。
- `google_identity`：Google 登录绑定。
- `apple_identity`：Apple 登录绑定。
- `phone_identity`：手机号验证码登录或找回账号。
- `wechat_identity`：中国区微信登录或小程序登录绑定。

同一个用户可以绑定多个登录方式，但这些登录方式都指向同一个 `user_id`。

这样做的好处：

- 用户换国家、换设备、换平台后仍然能找回账号。
- Web App、未来 iOS/Android App、小程序可以共用同一套用户资产。
- 中国区可以提供微信登录，但海外用户不会被微信挡住。
- 将来接入 LINE、Kakao 等地区化登录时，不会推翻原有账号系统。

## 6. 平台策略

### Web App / H5

第一阶段建议以 Web App / H5 为主。

原因：

- 活动链接可以被任何人打开。
- 适合邮件、社交平台、群聊、二维码传播。
- 更利于全球化和 SEO。
- 不被单一生态限制。

### 微信小程序

微信小程序适合作为中国市场增强渠道，而不是全球账号中心。

适合场景：

- 中国区活动组织者。
- 微信群传播。
- 快速微信授权登录。
- 微信支付或小程序内通知。

风险：

- 海外用户不一定有微信。
- 小程序内容很难自然进入全球搜索和跨平台分享。
- Web、App、小程序之间的账号打通需要额外设计。

结论：

GatherUp 应该先做全球可用的 Web 账号体系，再把微信作为中国区入口接入。

## 7. 验证码策略

需要发验证码的场景：

- 注册。
- 登录。
- 忘记密码。
- 绑定邮箱。
- 绑定手机号。
- 更换邮箱或手机号。
- 异常设备登录。

验证码建议规则：

- 6 位数字。
- 5-10 分钟有效。
- 同一邮箱或手机号限制发送频率。
- 同一 IP 限制发送频率。
- 验证失败次数过多后临时锁定。

## 8. 长期有效机制

正式系统应区分两类东西：

- 用户身份：数据库里的永久 `user_id`。
- 登录状态：短期 session 或 token。

建议：

- 数据库保存永久用户记录。
- 浏览器保存安全 session，不保存明文密码。
- session 到期后，通过 refresh token 或重新验证码恢复登录。
- 用户换设备时，通过邮箱、手机号、微信重新验证后找回同一个账号。

## 9. 用户数据模型

`users` 核心字段：

- `id`：系统内部永久 ID，不允许用户修改。
- `public_id`：GatherUp ID，用于展示和同行人填写。
- `public_id_change_count`：GatherUp ID 修改次数。
- `display_name`：昵称。
- `avatar_url`：头像。
- `email`：展示和联系用邮箱缓存，真实登录身份以 `user_auth_identities` 为准。
- `phone`：展示和联系用手机号缓存，真实登录身份以 `user_auth_identities` 为准。
- `wechat_openid`：微信身份缓存字段，未来可迁移到 `user_auth_identities.metadata`。
- `created_at`
- `updated_at`

`user_auth_identities` 核心字段：

- `id`
- `user_id`
- `provider`：`email`、`google`、`apple`、`phone`、`wechat`、`line`、`kakao`。
- `provider_user_id`：第三方平台返回的稳定用户标识，邮箱登录时可使用规范化后的邮箱。
- `email`
- `phone`
- `display_name`
- `avatar_url`
- `is_primary`
- `verified_at`
- `last_sign_in_at`
- `metadata`
- `created_at`
- `updated_at`

规则：

- `id` 是真正的数据关联字段。
- `public_id` 是展示 ID，不作为最终权限判断依据。
- `public_id` 最多允许修改两次。
- 同一个邮箱、手机号或第三方登录身份不能绑定多个用户。

## 10. 权限模型

权限不应该来自“账号类型”，而应该来自数据关系。

示例：

- 活动创建者可以编辑自己创建的活动。
- 报名用户可以查看自己的订单。
- 订单所属用户可以上传付款截图。
- 活动创建者可以确认或驳回该活动的付款。
- 付款确认后，订单所属用户可以选座。

未来如果需要多人协作，可以增加：

- 活动管理员。
- 付款审核员。
- 签到员。
- 只读协作者。

这些都应是活动级权限，不是全局账号类型。

## 11. v0.1 实施建议

第一步：

- 使用 Supabase Auth 建立邮箱登录。
- 创建 `users` 资料表。
- 首次登录后生成默认 GatherUp ID。
- 在页面上展示当前用户信息。

第二步：

- 加入验证码登录。
- 加入修改 GatherUp ID。
- 限制 GatherUp ID 最多修改两次。

第三步：

- 支持 Google 登录。
- 预留 Apple 登录。
- 支持手机号绑定。
- 预留微信登录入口，作为中国区增强。
- 增加账号安全设置页。
