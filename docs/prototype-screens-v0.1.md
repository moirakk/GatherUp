# GatherUp v0.1 原型页面说明

## 1. 目标

这份文档用于指导 GatherUp v0.1 的第一版界面原型。

它不追求覆盖未来所有功能，而是先把核心闭环画清楚：

活动广场 → 活动详情 → 登录报名 → 上传付款截图 → 等待审核 → 付款确认后选座 → 我的活动。

组织者侧闭环：

组织工作台 → 创建活动 → 活动管理台 → 审核报名/付款 → 管理座位 → 发布通知 → 导出名单。

## 2. 全局导航

### 桌面端

顶部导航：

- GatherUp 标识。
- 活动广场。
- 场地库。
- 我的活动。
- 组织工作台。
- 个人中心。

右侧：

- 创建活动按钮。
- 用户头像/菜单。

### 移动端

底部导航：

- 广场。
- 场地。
- 我的。
- 组织。

创建活动可以作为组织页内主按钮，不一定放在底部导航中。

## 3.1 场地库

### 页面目标

让组织者在创建活动前快速判断某个城市有哪些场地可能承接活动。

### 页面结构

顶部区域：

- 页面标题：场地库。
- 搜索框。
- 创建活动入口。

筛选区：

- 城市。
- 场地类型。
- 可办状态。

场地列表：

- 使用场地卡片。
- 桌面端 2 列。
- 移动端单列。

### 场地卡片

内容：

- 场地名称。
- 城市和区域。
- 场地类型。
- 可办状态。
- 适合活动标签。
- 容量备注。
- 组织者经验摘要。
- 评分。
- 最近验证时间。

### 场地详情页

内容：

- 场地名称。
- 可办状态。
- 适合活动。
- 价格备注。
- 联系建议。
- 沟通、环境、布置弹性、性价比评分。
- 优势。
- 注意事项。

视觉建议：

- 场地库不做成点评站风格。
- 信息密度要高，但保持卡片清晰。
- 强调“组织者经验”，不要只展示地址和评分。

## 3. 活动广场

### 页面目标

让参与者快速浏览公开活动，并进入详情页。

### 页面结构

顶部区域：

- 页面标题：活动广场。
- 搜索框。
- 筛选入口。

筛选区：

- 城市。
- 活动场景。
- 活动状态。
- 时间范围。

活动列表：

- 使用活动卡片。
- 桌面端可以 2-3 列。
- 移动端单列。

### 活动卡片

内容：

- 活动名称。
- 活动 ID。
- 活动场景和细分标签。
- 城市和时间。
- 场地名称。
- 单人价格。
- 剩余名额。
- 状态标签。

状态标签示例：

- 报名中。
- 即将截止。
- 付款确认中。
- 已满员。
- 已结束。

视觉建议：

- 卡片背景使用暖白或浅雾灰。
- 状态标签使用低饱和色。
- 不使用强烈营销风封面图。
- 如果暂时没有真实图片，可以使用低饱和色块和简洁图标。

## 4. 活动详情页

### 页面目标

让参与者在 10 秒内理解：这是什么活动、什么时候在哪、多少钱、还能不能报、下一步点哪里。

### 页面结构

头部信息卡：

- 活动名称。
- 活动 ID。
- 状态标签。
- 城市。
- 时间。
- 场地。
- 价格。
- 剩余名额。

关键行动区：

- 未登录：登录后报名。
- 已登录未报名：立即报名。
- 已报名未付款：去付款。
- 已上传截图待审核：等待审核。
- 已确认付款未选座：去选座。
- 已选座：查看我的报名。

详情区：

- 活动说明。
- 组织者说明。
- 报名规则。
- 付款规则。
- 最新通知。

### 文案原则

- 不解释过多平台功能。
- 每个状态只告诉用户下一步。
- 重要信息放在第一屏。

## 5. 登录/注册

### 页面目标

让参与者和组织者尽快进入流程。

### 登录方式

v0.1 设计上支持：

- 微信。
- 邮箱。
- 手机号。

### 原型呈现

优先显示：

- 微信登录。
- 邮箱登录。
- 手机号登录。

后续开发可以按成本决定先实现哪一种。

### 登录后

首次登录需要生成或设置：

- 昵称。
- GatherUp 公开 ID。

公开 ID 说明：

- 用于多人报名时被同行人填写。
- 最多可修改两次。

## 6. 报名页

### 页面目标

让参与者完成报名，并清楚看到付款下一步。

### 表单分组

报名信息：

- 昵称。
- 联系方式类型。
- 联系方式。

人数信息：

- 人数。
- 同行人 GatherUp ID。

备注：

- 给组织者的备注。
- 是否接受候补。

### 动态规则

如果活动不允许多人报名：

- 不显示人数选择。
- 系统默认人数为 1。
- 不显示同行人 ID。

如果活动允许多人报名：

- 显示人数选择。
- 显示每单最多人数。
- 当人数大于 1 时，显示同行人 ID 输入项。
- 同行人 ID 数量等于人数减 1。

### 提交后结果

报名成功卡片：

- 订单编号。
- 应付金额。
- 报名状态。
- 付款状态。
- 去上传付款截图按钮。

## 7. 付款截图页

### 页面目标

让参与者知道该付多少钱、怎么付、截图上传后现在处于什么状态。

### 页面结构

订单卡片：

- 订单编号。
- 活动名称。
- 报名人数。
- 应付金额。

付款说明卡片：

- 组织者填写的付款说明。
- 可放收款码图片位，v0.1 可先作为上传/链接占位。

上传区：

- 上传付款截图。
- 重新上传。
- 查看已上传截图。

状态区：

- 待上传截图。
- 已上传，等待组织者审核。
- 付款已确认，可以选座。
- 付款被驳回，请重新上传。

### 主要按钮

- 上传截图。
- 重新上传。
- 去选座。
- 返回我的报名。

## 8. 选座页

### 页面目标

让已确认付款的参与者完成座位选择。

### 前置状态

如果付款未确认：

- 不显示可操作座位图。
- 显示“付款确认后可选座”。
- 提供返回付款状态按钮。

### 页面结构

顶部订单卡片：

- 订单编号。
- 需要选择的座位数。
- 当前已选座位。

座位图：

- 屏幕方向提示。
- 座位状态图例。
- 座位矩阵。

底部固定操作：

- 已选座位摘要。
- 确认选座按钮。

### 座位状态

- 可选。
- 已选。
- 已被他人选择。
- 锁定不可选。

### 交互规则

- 已选数量必须等于报名人数才能提交。
- 多人订单可选多个座位。
- 单人订单只能选一个座位。
- 座位不要求连续。

- 即将开始。

- 报名状态。
- 付款状态。

下一步操作示例：

- 查看审核。
- 去选座。
- 查看订单。

绑定方式：

## 12. 创建活动页

### 页面目标

让组织者创建一场活动，同时不被过多字段压垮。

### 推荐布局

使用分组卡片：

1. 基础信息。
2. 报名规则。
3. 付款规则。
4. 订单规则。
5. 座位设置。
6. 活动说明。

### 基础信息

- 活动名称。
- 活动 ID。
- 主办 GatherUp ID。
- 协作组织者 GatherUp ID。
- 活动场景。
- 流程模板。
- 细分标签。
- 城市。
- 场地名称。
- 地址。
- 日期。
- 时间。

### 报名规则

- 活动人数上限。
- 报名截止时间。
- 是否允许多人报名。
- 每单最多人数。
- 是否接受候补。

### 付款规则

- 费用模式：免费活动/收费活动/AA 记账活动。
- 单人费用。
- 付款说明。

### 订单规则

- 编号格式。
- 编号前缀或活动代码。

建议编号格式：

- `GU-0001`
- `{eventCode}-0001`
- `{eventCode}-{YYYYMMDD}-0001`
- 自定义前缀 + 流水号。

### 座位设置

- 排数。
- 每排座位数。
- 不可用座位。

### 操作

- 保存草稿。
- 发布报名。

- 切换状态按钮。

### Tab

- 概览。
- 报名。
- 付款。
- 座位。
- 通知。
- 导出。

## 14. 管理台：概览

- 总报名数。
- 已付款人数。
- 待审核付款。
- 已选座人数。
- 剩余名额。
- 候补人数。

最近动态：

- 最新报名。
- 最新付款截图。
