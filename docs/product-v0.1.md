## 1. 产品定位

GatherUp 是一个面向线下活动组织者的流程管理工具，同时提供基础活动广场，让公开活动可以被浏览和报名。

第一版从线下观影这类流程清晰的活动切入，但产品边界不局限于粉丝活动。GatherUp 需要逐步覆盖校园活动、会议会务、好友聚会、工作坊、市集快闪等不同场景，避免组织者在问卷、群聊、表格、付款截图、座位表和签到工具之间反复切换。

v0.1 阶段，GatherUp 的核心仍然是活动组织工作台。用户登录后既可以参与活动，也可以创建活动。活动创建并发布后，可以进入活动广场，也可以通过链接分享给特定参与者。参与者先完成数调和地点投票，再进入报名、付款截图上传、选座和查看通知。

v0.1 界面先做中文。产品结构需要为后续扩展英文、日文、韩文等语言预留空间，避免之后重做信息架构。

## 2. 核心价值

对组织者：

- 集中收集报名信息。
- 不用在群聊里反复翻付款截图。
- 可视化管理座位。
- 导出干净的参与者名单。
- 减少重复私信和手动对账。

对参与者：

- 打开一个活动链接。
- 或从活动广场浏览公开活动。
- 登录或注册账号。
- 提交报名信息。
- 查看付款说明。
- 上传付款截图。
- 查看付款审核状态。
- 付款确认后选择座位。
- 查看活动通知。
- 查看自己历史参与过的活动。

## 3. 活动场景和第一版切入点

GatherUp 的活动类型采用“活动场景 + 流程模板”的设计。

活动场景包括同好活动、校园活动、会议会务、好友聚会、工作坊、快闪/市集等。

流程模板包括基础报名、报名收款、选座活动、签到活动、分时预约和记录型聚会。

第一版仍然可以从线下观影切入，因为它流程清楚、痛点集中：

1. 创建活动。
2. 分享活动链接。
3. 收集报名。
4. 收集付款截图。
5. 确认付款。
6. 开放选座。
7. 发布通知。
8. 现场签到。
9. 导出最终名单。

但在产品结构和数据模型里，不应该把“线下观影/生咖”写死成平台边界。它们应当被视为“同好活动”场景下的细分标签或模板组合。

## 4. 用户角色

### 组织者

组织者是第一优先级用户。

主要任务：

- 创建和配置活动。
- 配置数调时间选项。
- 配置地点投票选项。
- 上传或填写收款二维码/收款信息。
- 查看报名信息。
- 确认或驳回付款截图。
- 管理座位。
- 发布通知。
- 导出名单和付款记录。

### 参与者

参与者通过组织者分享的活动链接进入。

主要任务：

- 查看活动详情。
- 报名前登录。
- 参与数调，提交可参加时间。
- 参与地点投票。
- 提交报名。
- 获得报名编号/订单编号。
- 上传付款截图。
- 符合条件后选座。
- 查看报名状态和活动通知。

## 4.1 v0.1 已确认产品决策

- 参与者 v0.1 也需要账号。
- 登录方式以微信、邮箱、手机号为主。
- v0.1 界面语言先做简体中文。
- 多语言问题先放一放，但底层结构尽量不要阻碍未来扩展。
- v0.1 付款方式为“上传付款截图 + 组织者手动确认”。
- 组织者需要先填写活动信息、数调/地点投票配置和收款二维码，参与者才进入报名付款流程。
- 参与者正式报名前，需要先完成数调和地点投票。
- 公开活动页不暴露活动组织者入口，组织者必须从工作台进入管理。
- 同一登录用户对同一活动只能提交一份数调和一份地点投票，允许截止前修改但需要留记录。
- 活动处于“数调中”时，参与者只能提交调研和投票，不能提前生成订单、付款或占座。
- 参与者界面必须明确区分“意向已提交”“正式报名”“付款待确认”“已确认占位”等状态，避免误以为数调等于报名。
- 报名开放、付款确认、付款驳回等关键节点需要通知参与者。
- 订单详情页应作为参与凭证，展示订单号、参与人、付款截图、确认状态、选座/签到状态和退款规则。
- 每条报名都生成一个可见的订单编号，方便组织者和参与者查找。
- 订单编号格式由活动发起者在创建活动时选择。
- 参与者必须先完成付款确认，才能选座。
- 是否允许多人报名由活动组织者在每个活动里单独设置。
- 多人报名时，需要记录同行人的 GatherUp ID。
- 用户的公开 ID 仅允许修改两次。
- 每个登录用户都可以查看自己历史参与过的活动。
- 公开活动进入活动广场。

## 5. 活动生命周期

### 内部状态

- `draft`：草稿，活动还在准备中。
- `survey`：数调中，组织者正在收集意向或投票。
- `registration`：报名中，正式报名已开放。
- `payment`：付款确认中。
- `seating`：选座中。
- `confirmed`：已成团/已确认举办。
- `finished`：已结束。
- `cancelled`：已取消。

### v0.1 前台展示状态

为了让界面更简单，v0.1 可以先展示较少的状态：

- 草稿
- 数调中
- 待开放报名
- 报名中
- 付款确认中
- 选座中
- 已成团
- 已结束
- 已取消

数调和地点投票是 GatherUp 的前置核心流程，不应放到后续版本才考虑。

## 7. 主要页面

### 7.0 活动广场

用途：

展示公开活动，让参与者可以主动浏览和进入活动详情页。

v0.1 范围：

- 公开活动列表。
- 按城市筛选。
- 按活动场景筛选。
- 按活动状态筛选。
- 活动卡片显示名称、城市、时间、价格、剩余名额和状态。

暂不做：

- 个性化推荐。
- 社交动态。
- 复杂搜索排序。
- 用户关注关系。

### 7.1 组织者工作台

用途：

展示组织者创建的所有活动，以及当前最需要处理的事项。

关键内容：

- 活动卡片或活动表格。
- 活动状态。
- 报名人数。
- 已付款人数。
- 待确认付款人数。
- 未选座人数。
- 创建活动按钮。

主要操作：

- 创建活动。
- 进入活动管理台。
- 复制参与者报名链接。

### 7.2 创建活动页

用途：

用最少必要信息创建一场活动。

字段：

- 活动名称。
- 活动场景。
- 流程模板。
- 自定义活动标签。
- 城市。
- 场地/影院名称。
- 地址。
- 观影日期。
- 观影时间。
- 报名截止时间。
- 活动人数上限。
- 单人费用。
- 付款说明。
- 活动说明。
- 组织者联系说明。
- 活动可见性：进入活动广场/仅链接可见。
- 是否允许多人报名：是/否。
- 每单最多人数：仅当允许多人报名时显示。
- 订单编号格式。
- 订单编号前缀或活动代码：当所选编号格式需要时显示。

座位设置：

- 排数。
- 每排座位数。
- 可选的不可用座位。

主要操作：

- 保存草稿。
- 发布报名。

### 7.3 活动管理台

用途：

单个活动的核心组织者工作区。

建议分栏：

- 概览。
- 报名名单。
- 付款管理。
- 座位管理。
- 通知。
- 导出。

### 7.4 概览

用途：

给组织者一个快速运营视图。

关键指标：

- 总报名数。
- 已确认报名数。
- 待确认付款数。
- 未付款报名数。
- 已选座数。
- 剩余座位数。

实用模块：

- 活动公开链接。
- 当前活动状态。
- 下一步建议操作。
- 最近报名记录。

### 7.5 报名名单

用途：

管理参与者报名记录。

列表字段：

- 报名 ID。
- 订单编号。
- 参与者昵称。
- 联系方式。
- 人数。
- 报名状态。
- 付款状态。
- 选座状态。
- 提交时间。
- 备注。

报名状态：

- `pending`：已提交，待处理。
- `confirmed`：已确认。
- `waitlisted`：候补。
- `cancelled`：已取消。

操作：

- 确认报名。
- 移入候补。
- 取消报名。
- 添加组织者备注。
- 查看参与者详情。

### 7.6 付款管理

用途：

审核付款截图，并保持付款记录清楚。

列表字段：

- 订单编号。
- 参与者昵称。
- 应付金额。
- 付款截图缩略图。
- 付款状态。
- 提交时间。
- 确认时间。

付款状态：

- `unpaid`：未上传截图。
- `submitted`：已上传截图，等待组织者审核。
- `confirmed`：组织者已确认付款。
- `rejected`：付款截图被驳回，需要重新提交。
- `refunded`：已退款。

操作：

- 确认付款。
- 驳回付款。
- 标记退款。
- 添加付款备注。

v0.1 不接入官方支付。付款截图上传加组织者确认就够了。

### 7.7 座位管理

用途：

让组织者管理和查看座位分配情况。

座位状态：

- `available`：可选。
- `held`：临时占用。
- `assigned`：已分配给参与者。
- `blocked`：不可选。

组织者操作：

- 锁定或解锁座位。
- 给参与者分配或调整座位。
- 清空座位。
- 打开公开选座链接。

v0.1 座位图可以先通过“排数 + 每排座位数”自动生成，不需要支持复杂影院座位图导入。

### 7.8 通知

用途：

创建活动通知，参与者可以在报名页查看，组织者也可以复制到群聊。

通知类型：

- 报名通知。
- 付款提醒。
- 选座通知。
- 成团通知。
- 活动当日通知。

字段：

- 标题。
- 正文。
- 可见状态。
- 发布时间。

操作：

- 发布。
- 保存草稿。
- 复制文案。

### 7.9 导出

用途：

给组织者导出可在外部使用的数据。

导出选项：

- 报名 CSV。
- 付款 CSV。
- 座位分配 CSV。

最低限度 CSV 字段：

- 报名 ID。
- 订单编号。
- 昵称。
- 联系方式。
- 人数。
- 报名状态。
- 付款状态。
- 座位号。
- 备注。

## 8. 参与者页面

### 8.0 参与者个人中心

用途：

让登录后的参与者查看账号信息和历史活动。

内容：

- 用户公开 ID。
- ID 剩余修改次数。
- 绑定的登录方式：微信、邮箱、手机号。
- 历史参与活动。
- 当前待处理事项：待付款、待审核、待选座。

规则：

- 公开 ID 最多只能修改两次。
- 历史参与活动包括报名过、付款过、候补过和已完成的活动。

### 8.1 公开活动详情页

用途：

展示活动关键信息，并引导参与者报名。

内容：

- 活动名称。
- 活动状态。
- 日期和时间。
- 场地。
- 价格。
- 剩余名额。
- 活动说明。
- 组织者说明。

主要操作：

- 登录并报名。

次要操作：

- 查看我的报名。

### 8.2 报名表

访问规则：

参与者必须登录后才能提交报名。

字段：

- 昵称。
- 联系方式类型。
- 联系方式。
- 人数：仅当组织者允许多人报名时显示。
- 同行人 GatherUp ID：仅当组织者允许多人报名且人数大于 1 时显示。
- 给组织者的备注。
- 是否接受候补：是/否。

提交后：

- 生成订单编号。
- 显示报名状态。
- 显示应付金额。
- 显示付款说明。

订单编号规则：

- 订单编号对参与者和组织者都可见。
- 订单编号需要出现在付款页、组织者付款审核表、参与者状态页和导出表中。
- 活动发起者可以选择订单编号格式。
- 可选格式示例：`GU-0001`、`{eventCode}-0001`、`{eventCode}-{YYYYMMDD}-0001`、自定义前缀加流水号。
- 如果活动不允许多人报名，人数固定为 1。
- 如果活动允许多人报名，人数不能超过组织者设置的每单人数上限。
- 多人报名时，系统需要记录同行人的 GatherUp ID。

### 8.3 付款截图页

内容：

- 订单编号。
- 应付金额。
- 付款说明。
- 上传付款截图。
- 付款审核状态。

状态：

- 等待上传付款截图。
- 等待组织者审核。
- 付款已确认。
- 付款被驳回。

付款规则：

- v0.1 不接入官方支付。
- 参与者上传付款截图或付款凭证。
- 组织者手动确认或驳回。
- 付款确认前不能选座。

### 8.4 选座页

访问规则：

参与者只有在付款确认后才能选座。

内容：

- 座位图。
- 当前已选座位。
- 座位状态说明。

规则：

- 可选座位数必须等于报名人数。
- 已分配座位不能被其他人选择。
- 锁定座位不能选择。
- 如果活动允许多人报名，一个订单可以选多个座位。
- 如果活动不允许多人报名，一个订单只能选一个座位。

### 8.5 我的报名页

用途：

让参与者在一个地方查看自己的报名状态。

内容：

- 活动摘要。
- 订单编号。
- 报名状态。
- 付款状态。
- 座位信息。
- 最新通知。
- 签到码占位。

签到码可以后续再加。

## 10. v0.1 建议开发顺序

### 第一阶段：项目骨架

- 项目初始化。
- 基础应用布局。
- 中文文案结构。
- 组织者工作台。
- 活动广场基础列表。
- 创建活动表单。
- 活动管理台外壳。

### 第二阶段：登录和报名闭环

## 11. 设计方向

GatherUp 应该像一个安静、可靠的活动运营工具，而不是粉丝社区首页。

界面原则：

- 优先显示状态和下一步操作。
- v0.1 使用中文界面文案。
- 审美向 iOS 看齐：轻盈、克制、清楚、顺手。
- 主色调采用莫兰迪色系，整体低饱和、柔和但不灰脏。
- 可以多使用卡片承载信息，但卡片用于活动条目、表单分组、状态面板等明确内容，不做过度堆叠。
- 表格要清晰可读。
- 参与者页面优先适配手机。
- 组织者页面可以信息密度高一些，但不能混乱。
- 颜色克制，用清楚的状态色辅助判断。
- App 内部不要做过重的营销式页面。
- 留白、圆角、阴影和动效都要轻，不做廉价感强的渐变和高饱和装饰。
- 图标风格统一，按钮和状态反馈接近 iOS 的清晰度。

视觉关键词：

- 高级审美。
- 莫兰迪色系。
- iOS 感。
- 简洁。
- 高效。
- 温和但专业。

建议状态色：

- 草稿：中性灰。
- 开放中：蓝色或绿色。
- 待处理：琥珀色。
- 已确认：绿色。
- 已取消/已驳回：红色。
- 已结束：弱化的中性色。

## 12. 待确认问题

- 微信登录在 Web 版和未来小程序版中的实现优先级。
- 手机号登录是否从 v0.1 就接入短信验证码，还是先用邮箱验证码降低成本。
- 同行人 ID 是否必须对应已注册用户，还是允许先填写后邀请绑定。
- 订单编号格式的第一批可选模板具体定几种。
- 活动广场是否需要审核机制，避免公开活动质量失控。
